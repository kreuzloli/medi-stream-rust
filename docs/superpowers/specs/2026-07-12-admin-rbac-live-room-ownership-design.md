# 管理员 RBAC 与直播间归属扩展设计

## 目标

在共享数据库中增加独立管理员账号和 RBAC 表，使管理员可以作为直播间创建者。同时扩展直播间的科室、疾病分类和置顶能力，并同步调整本项目已有的 `user_info`、`live_room` 模型与接口。

## 范围

本项目负责：

- 在 `db/medi.sql` 尾部追加增量 SQL。
- 增加管理员、角色、权限及其多对多关联表。
- 调整本项目已有的 `user_info` 和 `live_room` 数据模型、Repository、Service、接口、注释及必要日志。
- 保持现有普通用户创建和管理直播间的能力。

本项目不负责：

- 管理员登录、鉴权和令牌签发。
- 管理员、角色、权限的后台 CRUD 接口。
- 后台管理路由和页面。

这些能力由共享同一数据库的后台项目实现。

## 数据库设计

### 管理员与 RBAC

新增以下表：

- `administrator`：独立保存管理员登录名、密码哈希、姓名、状态、软删除标记、最后登录时间和审计时间。
- `admin_role`：保存角色编码、名称、描述、状态和审计字段。
- `admin_permission`：保存权限编码、名称、资源类型、描述、状态和审计字段。
- `administrator_role`：管理员与角色的多对多关联表。
- `role_permission`：角色与权限的多对多关联表。

管理员账号不复用 `user_info` 或 `user_login_account`。关联表使用联合唯一约束避免重复授权，并为反向查询列建立索引。

### 直播间所有者

保留 `live_room.owner_user_id`，但通过增量 SQL 将其调整为可空。新增可空的 `owner_admin_id`：

- 普通用户创建时只填写 `owner_user_id`。
- 管理员创建时只填写 `owner_admin_id`。
- 两个字段必须且只能有一个非空。
- 两个字段分别使用外键关联 `user_info.id` 和 `administrator.id`。

现有直播间数据已经具有 `owner_user_id`，因此调整后仍满足所有者约束，不需要迁移所有者数据。

### 直播间分类与置顶

为 `live_room` 增加：

- `department_id BIGINT UNSIGNED NULL`，关联 `department.id`。
- `disease_id BIGINT UNSIGNED NULL`，关联 `disease.id`。
- `is_top TINYINT NOT NULL DEFAULT 0`，`0` 表示不置顶，`1` 表示置顶。

科室和疾病均为可选。数据库分别保证引用记录存在；疾病与科室的从属一致性由保存或更新直播间的业务逻辑校验，因为普通外键无法表达“所选疾病必须属于所选科室”的跨列规则。两者同时填写时必须匹配；只填写其中一个时允许保存。

不增加额外的直播间类型枚举字段，`department_id` 和 `disease_id` 本身表达业务分类。

## 增量迁移策略

保持现有 `CREATE TABLE` 语句不变，在 `db/medi.sql` 末尾按依赖顺序追加：

1. 创建管理员及 RBAC 表。
2. 删除 `live_room.owner_user_id` 的原外键。
3. 将 `owner_user_id` 调整为可空。
4. 增加管理员所有者、分类和置顶字段。
5. 重新增加普通用户所有者外键，并增加管理员、科室、疾病外键。
6. 增加所有者互斥检查约束及查询索引。

增量 SQL 面向当前项目使用的 MySQL 8。迁移应在应用发布前执行，并在执行前确认生产库不存在与新增表、字段或约束同名的对象。

## 本项目代码调整

先通过 CodeGraph 定位所有 `user_info` 与 `live_room` 相关模型和调用链，再做最小范围修改：

- `LiveRoom`、保存请求和分页查询模型增加管理员所有者、科室、疾病和置顶字段。
- Repository 的插入、更新、详情和分页 SQL 同步新增字段。
- Service 校验两个所有者字段必须二选一。
- 当科室和疾病同时填写时，校验疾病属于该科室。
- 分页默认按 `is_top DESC, id DESC` 排序，使置顶直播间优先展示；相同置顶状态下保持原有倒序规则。
- 如果 `user_info` 相关 DTO 或接口包含直播间所有权信息，同步适配双所有者结构；不改动与本需求无关的用户接口。
- 更新字段注释和非显而易见的业务约束注释。
- 校验失败日志包含直播间 ID、普通用户 ID、管理员 ID、科室 ID 和疾病 ID 中可获得的业务标识；不记录密码哈希等敏感信息。

现有公开接口路径保持不变。请求和响应只增加新字段，不重命名或删除已有字段。

## 错误处理

- 两个所有者均为空：返回参数校验错误。
- 两个所有者同时存在：返回参数校验错误。
- 所有者、科室或疾病不存在：由业务校验返回明确错误；数据库外键作为最终一致性保护。
- 科室与疾病同时存在但不匹配：返回参数校验错误。
- 数据库迁移失败：保留原始数据库错误，不继续执行依赖该结构的应用发布。

## 验证

- 在空库执行完整 `db/medi.sql`，确认原建表与尾部增量 SQL 可连续成功执行。
- 在已有旧结构和普通用户直播间数据的数据库执行增量部分，确认数据保留且所有者约束成立。
- 验证普通用户和管理员分别可以创建直播间。
- 验证双所有者、无所有者以及科室疾病不匹配请求被拒绝。
- 验证分类字段可单独为空或单独填写。
- 验证置顶直播间在分页结果中优先，并保持同组内原有倒序。
- 对调整后的 Rust 代码执行格式化、编译检查和相关测试。

## 兼容性与注意事项

- `owner_user_id` 从必填变为可空，Rust 模型必须改为可选类型，否则管理员直播间查询会反序列化失败。
- 新增字段具有空值或默认值，现有普通用户直播间数据不需要回填。
- 管理员表只提供共享数据库契约，后台项目应自行实现密码策略、登录限流、鉴权和权限缓存。
- 置顶排序会改变直播间分页顺序，这是本次需求期望的可见行为变化。
